Streaming Deadlock Fix (2026-02-22):
    Root cause: iter_lines() on streaming HTTP has no read timeout; control_queue.get()
    had no timeout → both the control stream process and main loop deadlocked when
    Lichess's event stream went silent after a game ended/aborted.
    Fix (lichess-bot-master/lib/):
      lichess.py: timeout=(15, 60) on get_event_stream() and get_game_stream()
                  (tuple = connect/read; 60s read timeout matches Lichess ~30s keepalive)
      lichess_bot.py: ReadTimeout/RemoteDisconnected/ChunkedEncodingError/ConnectionError
                      caught in watch_control_stream() → log + reconnect instead of crash
      lichess_bot.py: control_queue.get(timeout=30) in next_event() → handles Empty so
                      main loop wakes up every 30s even if stream is stalled
    Watchdog: /root/scripts/watchdog_lichess_bot.sh (cron every 15 min) — restarts bot
    if systemd service is active but journalctl silent >30 min (belt-and-suspenders)

Feature Candidates:
    Ranked by estimated Elo gain. Ranges are rough — small samples have wide CIs.
    Already done: Texel tuning (v20), pondering (v21), clock differential TM (v22),
    razoring, history-adjusted LMR, LMP, king tropism, passed pawn king proximity,
    candidate passed pawns, mop-up eval, threat detection (v19);
    Mate Distance Pruning (v23, +11 Elo), ETT cache (v23, +43 Elo combined w/MDP),
    Endgame scaling opp-color bishops + lone rooks (v23, Texel-tuned to 93/109/0).
    Tested + skipped: Free Passer (MSE worse, doubles PSTs), Correction History
    (46% game test, regression), Safe Check Bonuses (MSE flat), Virtual Mobility
    (MSE marginally worse).

    --- Search ---

    Mobility eval                          ~20-40 Elo
        Count legal moves for knights, bishops, rooks, queens (excluding
        squares attacked by enemy pawns and own pieces). Add a bonus per
        mobility point, scaled by piece type. Simbelmyne uses ranges of
        0-99cp (N/B/R) and 0-300cp (Q). This is the single biggest missing
        eval term vs stronger engines.

    Correction history                     ~20-30 Elo  [TESTED v23: 46% game test, SKIP]
        Five hash tables that adjust static eval based on position features:
        pawn structure (pawn hash), non-pawn material by color, minor pieces,
        overall material, and continuation moves. Each table is a running
        average, weighted more for deeper searches. Applied before any
        pruning decisions. +46 Elo for pawn correction alone in Simbelmyne.
        Note: tested with simple EMA approach (±8cp cap); regressed vs v22.
        May need more careful implementation to see gains.

    Virtual mobility (king danger)         ~5-10 Elo  [TESTED v23: MSE +0.000003, SKIP]
        Compute the queen's mobility from the king's square (how many squares
        a queen could reach if placed on the king's square). Larger = more
        open = more dangerous. Subtract as a king safety penalty, MG only.
        Complements the existing attack-count penalty.

    Safe/unsafe check bonuses in eval      ~5-10 Elo  [TESTED v23: MSE flat +0.00008, SKIP]
        Bonus for opponent having safe checks available against our king
        (checks that can't be recaptured). Separate bonuses for sliding piece
        checks vs knight checks. Used as part of king danger scoring.

    Endgame scaling for drawn endings      ✅ DONE v23 (~MSE -0.00017, ~+3 Elo)
        opp-color bishops: 93/128 = 72.7%; lone rooks: 109/128 = 85.2%
        Texel-tuned; pawn multiplier tuned to 0 (no pawn adjustment).

    Mate Distance Pruning                  ✅ DONE v23 (~+11 Elo vs v22)
        Tighten alpha/beta bounds based on the shortest possible mate. If we
        can't improve on a mate-in-N already found, prune immediately.
        alpha = max(alpha, -MATE + ply); beta = min(beta, MATE - ply).
        If alpha >= beta, return alpha. Zero overhead — just two comparisons
        before the search loop.

    Quiescence checks (depth -1)           ~3-7 Elo
        In quiescence, at the first ply (ply immediately after dropping from
        negamax), generate non-capturing checks in addition to captures. Cheap
        checks that can be verified with SEE >= 0 only. Lets the engine see
        forcing quiet checks that switch from -cp to 0 (drawing back). Adds
        complexity; gate on depth == -1 to avoid deep explosion.

    Staged move generation                 ~3-6 Elo (speed)
        TT pre-try is in v18 already. Next: separate captures from quiets in
        generate_moves(). If a capture causes a beta-cutoff, never generate
        quiets at all (~20-30% of internal nodes saved). Requires splitting
        generate_moves() into generate_captures() + generate_quiets() and
        calling them separately in the move loop.

    Multi-cut pruning                      ~3-6 Elo
        At non-PV nodes depth>=6, not in check: try up to M=6 moves with a
        null window at depth - R (R=3). If C=3 of them fail high (>= beta),
        the node is almost certainly a cut node — return beta immediately.
        Avoids wasting time at All-nodes. Gate: not already in null-move search,
        score bounds not mating. Can false-prune; tune M/C/R conservatively.

    Recapture Extension                    ~3-6 Elo
        Extend by 1 ply when the current move is a recapture on the same square
        as the previous move (opponent's last capture is being answered by
        recapture). Identifies forcing tactical lines. Gate: non-PV or PV both
        work; typically limited to depth >= 3 to avoid deep explosions.

    Stronger threat detection              ~3-5 Elo  [PARTIAL DONE v24: minor→heavy KEPT (MSE Δ−0.000003); pawn-push SKIP (worse)]
        v19 has pawn->piece threats (8cp/undefended). Extend to:
        (a) minor piece attacks on undefended enemy heavy pieces: KEPT in v24 (tp[780]=15,
        MSE 0.17697393→0.17697073, Δ−0.000003). Knights/bishops attacking enemy rooks/queens
        not defended by enemy pawns.
        (b) pawn push threats (bonus for pawn that can safely advance to attack enemy piece):
        TESTED, MSE 0.17699 (worse), SKIP.

    Pawn push threats                      ~3-5 Elo  [TESTED v24: MSE 0.17699 (worse), SKIP]
        Bonus for pawns that can safely push to attack undefended enemy
        pieces (not just attacks from current square as in v19's threat
        detection). Encourages aggressive pawn advances.

    Space advantage                        ~2-5 Elo  [TESTED v24: MSE 0.17699 (worse), SKIP]
        Count squares in extended center (c3-f5 white, c4-f6 black) not attacked
        by enemy pawns and with own pawn on same file. Bonus = count * TP_SPACE,
        MG-scaled. MSE worse — likely duplicates PST center bonuses.

    Improving heuristic for LMP            ~2-4 Elo
        "improving" = static_eval > eval two plies ago. If improving, be less
        aggressive with LMP (higher threshold). Requires storing static_eval
        per ply: static_evals[ply]. Gate existing LMP threshold on improving flag.
        Change: lmp_base = improving ? lmp_threshold[depth]*2 : lmp_threshold[depth].

    Passed Pawn Extension                  ~2-4 Elo
        Extend by 1 ply when a passed pawn advances to rank 6 (White) / rank 3
        (Black). These advances are often critical and under-searched. Gate:
        only in non-check positions to avoid compounding with check extension.
        Can be 0.5-ply (via fractional extensions) or full 1-ply.

    Pawn push evaluation                   ~2-4 Elo
        Bonus for pawns that can safely advance (no enemy pawn attack on
        destination, no pin). Encourages space-gaining pawn breaks. Simple:
        for each pawn, check if stop_sq is safe (not attacked by enemy pawns)
        and add a small bonus (3-5cp). Scale by advancement.

    Root move ordering by nodes searched   ~2-4 Elo (time management)
        After each iteration of iterative deepening, sort root moves by the
        number of nodes they generated (descending). Moves that consumed more
        nodes tend to be more critical. Improves move ordering on the next
        iteration and gives a signal for "bestmove stability" time management.

    Internal Iterative Reduction (IIR)     ~2-4 Elo  [SKIP: pure search change, needs game test]
        At non-PV nodes with depth >= 4 and no TT move: reduce depth by 1
        instead of running a shallow IID search. Simpler and cheaper than IID
        (no extra search). Our IID runs a depth-2 reduced search at PV nodes;
        IIR just subtracts 1 from depth at non-PV nodes with no TT hit.
        Complementary: keep IID for PV nodes, add IIR for non-PV. No new params.

    Separate PV/non-PV LMR                ~1-3 Elo  [SKIP: pure search change, needs game test]
        PV nodes currently get the same LMR reductions as non-PV. PV nodes
        deserve less aggressive reduction. Split: non-PV step keeps current
        rate, PV uses a higher step (e.g. half the reduction growth) → ~1 fewer
        ply on average at PV. Gate on is_pv flag already present in negamax.
        Two separate reduction formula branches.

    LMP score filter                       ~1-2 Elo  [SKIP: pure search change, needs game test]
        Current LMP skips all quiet moves after lmp_threshold[depth] quiets
        regardless of move history score. Better: also require move_score < -65
        (only prune clearly-bad-history moves). Prevents LMP from discarding
        late-ordered moves that happen to have strong history scores. Add one
        condition to the existing LMP check.

    --- Evaluation ---

    Free Passed Pawn bonus                 ~3-6 Elo  [TESTED v23: MSE worse at all coefficients, SKIP]
        Beyond the basic passed pawn bonus: detect "free" passed pawns —
        passed pawns with no enemy pieces (not just pawns) in front of them
        on the file. Reward these significantly more since they can promote
        without tactical obstacles. Applies a large MG+EG bonus for free
        passers separate from the base passed pawn score.

    Pawn chain / connected pawns bonus     ~2-4 Elo
        Bonus for pawns that are directly connected (on adjacent files,
        one rank apart — i.e., one pawn defends the other). Encourages
        solid pawn chains vs isolated pawn weaknesses. ~10-15cp per connected
        pair. Our isolated/doubled penalties already penalize disconnected pawns;
        this adds positive incentive.

    Center control bonus for pawns         ~2-4 Elo  [TESTED v24: MSE 0.17701 (worse), SKIP]
        Bonus for pawns on central squares (d4/e4 white, d5/e5 black) — beyond
        what PSTs already give. Explicit center occupation bonus. MSE worse —
        PSTs already capture center pawn value well enough.

    Castling rights evaluation penalty     ~2-4 Elo
        Eval penalty (MG only) for having lost castling rights without
        having castled. If king has not castled and castling rights for
        both sides are gone, apply a penalty (e.g., -30cp MG). Encourages
        the engine to castle early and punishes opponents who can't.
        Simple to implement: check (castling_rights == 0 && !has_castled).

    Connected knights bonus                ~1-3 Elo
        Bonus when two knights are on adjacent squares (can defend each
        other). ~10cp MG bonus for connected knight pairs.

    Passed pawn bonus indexed by rank      ✅ DONE v24 (MSE 0.17733→0.17700, Δ−0.00033)
        We have flat PASSED_MG/PASSED_EG scalars (Texel-tuned to 30 MG).
        Replace with 8-entry rank-indexed arrays PASSED_PAWN_MG[8] and
        PASSED_PAWN_EG[8]. Rank-specific values let the tuner learn the true
        curve — rank-6/7 passers deserve exponentially more reward than rank-3.
        Reference shape: [(11,-33), (16,17), (19,55), (30,59), (41,50),
        (22,41), (25,14), (37,26)] for ranks 0-7 (MG,EG pairs). The pawn eval
        already has rank = sq >> 3 available. Remove flat PASSED_MG/PASSED_EG
        from tp[], add 16 new entries. Big Texel tuning target.

    Split backward pawn: open vs closed    ~2-5 Elo  [TESTED v24: MSE 0.17740 (worse), SKIP]
        We have a single BACKWARD_PAWN penalty. Split into two:
          BACKWARD_PAWN_OPEN: harsher (~20-25cp MG) — exposed to rook attacks
          BACKWARD_PAWN_CLOSED: milder (~10-15cp MG) — blocked by enemy pawn
        Open file = no pawn (either color) ahead on that file; use existing
        file_masks. Two new tp[] params in pawn_eval_side(), Texel-tunable.

    Chained pawn bonus indexed by rank     ~2-4 Elo  [TESTED v24: MSE 0.17714 (worse), SKIP]
        Flat connected-pawn bonus tested and reverted in v23 (flat=wrong).
        Better: rank-indexed array CHAINED_PAWN[8] where advancing connected
        pawns get exponentially more reward. E.g. init [0,1,10,19,26,32,40,0]
        cp (rank 0/7=0, rank 5/6≈40cp). Detection: pawn_attacks[color][sq] &
        own_pawns_bb (same as our failed flat test). The rank-indexed version
        is meaningful in endgames where advanced chain passers are decisive.
        8 MG + 8 EG new tp[] params.

    Inner/outer mobility weighting         ✅ DONE v24 (MSE 0.17700→0.17697, Δ−0.00003)
        Current mobility counts all safe squares equally. Better: weight
        "inner" (central) squares more than "outer" (rim) squares. Inner =
        c3–f6 area (16 central squares), precomputed as INNER_SQUARES mask.
        For each piece type: inner_mob and outer_mob counted separately with
        different per-piece multipliers (inner ~3-11cp, outer ~2-3cp).
        Replaces current isqrt_x20 mobility formula. 8 new tp[]
        params (N/B/R/Q × inner/outer), all Texel-tunable.

    King area threats as tunable array     ✅ DONE v24 (MSE 0.17735→0.17733, Δ−0.00002)
        We use penalty = min(danger*danger/4, 150) (quadratic formula). Replace
        with an 8-entry indexed array KING_AREA_THREATS[8] (threat_count 0-7 →
        penalty), Texel-tunable. Lets the tuner find the true non-linear shape.
        Keep existing attacker-weight accumulation (Q=5, R=3, B=2, N=2, P=1)
        → clamp to [0,7] → look up array. Reference shape: roughly
        [0, 0, 0, 0, 0, 40, 85, 165] cp (exponential growth). 8 new tp[]
        MG-only params (king danger is MG-only in our engine).

    --- Infrastructure ---

    Evaluation TT cache (ETT)              ✅ DONE v23 (~+43 Elo combined w/ MDP)
        128K-entry cache keyed on hash_key, stores white's perspective score.
        Retrieve: (side==white) ? ee->score : -ee->score. Cleared on ucinewgame.
        Excluded from TUNER mode (#ifndef TUNER guards).

    UCI multipv                            0 Elo (analysis only)

        and debugging; zero effect on game strength.
